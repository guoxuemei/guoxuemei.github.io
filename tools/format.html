# 多人项目代码格式化

# husky 与 lint-staged 和prettier 的使用

多人开发时,不同人使用的编辑器缩进不同,代码风格不一致等等,导致提交的代码错乱无序,在有代码对比的场景下,非常影响对比视觉

可以使用git hooks 提供的pre-commit,在提交前,把暂存区的内容进行format再提交,这样仓库的代码,就会保持同一个风格 而git默认的hooks存在于.git/hooks文件夹,这个文件夹是不进行版本控制的,在本地生效的pre-commit无法提交被别的同事同步到 因此使用husky工具,该库核心是修改git的core.hooksPath为项目中的.husky/_,而这个目录,每个同事都可以通过执行husky命令初始化得到

在pre-commit中,就可以执行format相关指令了

prettier只能格式化本地代码,无法格式化已经存入暂存区的代码,lint-staged就可以解决这个问题 核心是:lint-staged可以针对暂存区的代码,使用特定指令格式化,然后把格式化以后的代码,提交至git,并且可以把格式化后的代码放回本地工作区,这样本地的该文件,也保持统一风格了

1 安装开发依赖

npm i -D husky lint-staged prettier

# 完成后 执行如下命令
#1. 修改git的core.hooksPath
#2. 添加一个pre-commit hook文件
#3. package.json 的script中,会多一条 "prepare": "husky"
npx husky init

这个图里使用的是npx husky, 是初始化.husky/_目录的, npx husky init 除了初始化这个目录,还会在该目录同级下,自动创建一个pre-commit,也就是下面的第四步操作

2 项目根目录创建prettier配置文件,简单设置了几个常用规则,我用的.prettierrc.js格式,方便写注释

module.exports = {
  // 语句末尾不加分号
  semi: false,
  // 不使用tab缩进(但是tab键行为会替换为空格缩进,缩进数为tabWidth)
  useTabs: false,
  tabWidth: 2,
  // 集合尾部逗号[1, 2, 3, 4,]=>[1, 2, 3, 4]
  trailingComma: 'none',
  // 字符串使用单引号还是双引号,这里是使用单引号
  singleQuote: true
}
  • prettier也可以设置忽略文件.pretierignore,用于忽略不需要格式化的文件/文件夹,如果项目有.gitignore,也会一并读取该文件配置内容

3 项目根目录新建lint-staged配置文件,添加格式化规则 .lintstagedrc.json

{
  "*.{js,ts,vue}":"prettier --write",
  "*.{css,scss,less}":"prettier --write",
  "*.{html,htm}":"prettier --write"
}
  • 注意: 这里的ts,vue,scss等前面,不要加空格,否则匹配不到该格式

4 .husky目录下,新建pre-commit文件,添加命令npm run lint:staged,该文件需要上传git,用于其他同事拉取

package.json 中配置这么两条指令

"scripts": {
  "prepare": "husky",
  "lint:staged": "lint-staged"
},

其中,prepare是执行完npx husky init以后,自动给添加的,如果没有就手动添加一下

prepare是个特殊的npm钩子,在npm install(不带参数)命令执行完以后自动执行

同事拉取到代码以后,如果不执行npm install,那么手动执行一下prepare指令,就可以正常使用了
